SharkBay Teamwork Feature Design

Teamwork 的目标是让小团队成员各自电脑里的 agent 共享足够的项目工作上下文。 MVP 采用 local-first 文件协议：清晰的本地 instruction、Markdown task files、 completed task records，以及一个统一的 Tasks UI。

Status: simplified proposal Branch: teamwork Date: 2026-05-15 MVP: local file protocol
Core Decision

SharkBay teamwork harness 在 MVP 中由 local instructions + Markdown task files + team task records 组成。 Agent 负责理解用户意图并维护 task files；SharkBay 负责读取这些文件、展示 Tasks UI，并把完成的 task record 同步到 team context branch。 

MVP building blocks:
- local ignored instruction files
- local Markdown task files
- optional root-level ignored instruction adapters
- completed task records in context branch
- Tasks UI from local and team Markdown

Management Model

团队工作中的 agent 输出属于工作职责的一部分，默认可以在团队内透明。 共享的是 task-level 工作摘要、变更结果、验证和 commit evidence。各家 CLI 的 session/transcript 继续由对应 client 自己管理。

  • GitHub 身份和 repo 权限复用 gh。
  • GitHub Issues 可以作为 task 的来源或背景。
  • 重要产品和开发决策仍进入 docs/。
  • Shared context 只保存适合团队审计和复用的工作记录。

Identity Model

Teamwork 写入使用 gh 验证本机 GitHub 身份和 repo 权限， 成员身份复用 GitHub Collaborators。

  • gh api user 提供 GitHub login、user id 和 avatar URL。
  • gh api repos/OWNER/REPO/collaborators/LOGIN/permission 提供 repo permission。
  • machineId 是本机本地生成的短 id，用于区分同一用户多台电脑。

Architecture

1. Install 当前 worktree 生成本地 .sharkbay/，并把 harness 文件加入 local git ignore。
2. Instruct SharkBay 通过启动 prompt、root ignored instruction file，或 UI 复制文本告诉 agent 读取 harness protocol。
3. Work Agent 按 protocol 工作并维护当前 task Markdown；SharkBay 读取并展示。
4. Finish Task 结束时在同一个 Markdown 文件里写 summary、verification、files 和 commit。
5. Sync Completed task record 写入 sharkbay-team-context， 并刷新本地 .sharkbay/team-context/ mirror。
Harness Boundary

并发和 dirty file 风险通过 instruction、Tasks UI 提醒、Git status、 commit review 和 LLM 自律来处理。SharkBay 的 guard 重点放在防止本地 teamwork harness 文件被提交。

Instruction Discovery

SharkBay 为每个 agent client 准备本地 instruction 入口。支持启动 prompt 的 client 可以直接收到 bootstrap；偏好根目录约定文件的 client 可以读取本地 ignored root adapter。

  1. 优先：client 支持启动 prompt，SharkBay 传入一句 bootstrap。
  2. 次选：生成本地 ignored root instruction file，让 client 自己发现。
  3. 最后：UI 展示一条可复制 instruction，由用户手动发给 agent。

AGENTS.md Adapter

Root instruction file 是本地 adapter。SharkBay 只管理自己带 marker/checksum 生成的文件；已有同名 tracked 或 untracked 文件时进入用户确认流程。

# SharkBay Local Agent Entry

This worktree uses SharkBay Teamwork.

Before making persistent project changes, read:

`.sharkbay/harness/protocol.md`

If the protocol file exists, follow it for task file naming, frontmatter,
status updates, verification notes, and sync readiness.

If the protocol file is missing or unreadable, ask the user whether to continue
without SharkBay task tracking.

protocol.md

Protocol 是 agent 维护 task Markdown 的 contract。它给 LLM 足够具体的模板和 状态规则，同时把业务判断留在 agent client 内。

## Agent Responsibility

You maintain SharkBay task files directly.
SharkBay reads and displays them.

## Team Context

Synced task records from the team are available in:
.sharkbay/team-context/tasks/**/*.md

Treat .sharkbay/team-context/ as read-only. Write only your own task records
under .sharkbay/tasks/.

Before editing files, making design decisions, or continuing work that may
overlap with prior team work, search the team context mirror.

Useful searches:
- rg "browser fullscreen" .sharkbay/team-context/tasks
- rg "docs/shared/teamwork-design.html" .sharkbay/team-context/tasks
- rg "issue #123" .sharkbay/team-context/tasks

If a previous task affects the current work, read that task record and mention
its taskId in the current task's Work Summary or Notes For Future Agents.

## When To Create Or Update A Task

Create or update a SharkBay task file before performing a persistent
project-changing unit of work.

Project-changing work includes:
- editing files
- generating a persisted project artifact
- running a multi-step implementation or verification workflow
- preparing a commit
- turning an accepted issue or request into local work

Discussion, explanation, code reading, and design exploration can remain outside
task tracking until the user asks to record it or implementation begins.

## Task Identity

Use one task file per logical task.

- taskTag: 6 uppercase characters, for example A7K3P9
- taskId: <taskTag>-u<githubUserId>-m<machineId>
- filename: .sharkbay/tasks/<taskId>-<slug>.md
- slug: short lowercase words joined with hyphens

Example:
.sharkbay/tasks/A7K3P9-u3960864-m7f3a6-update-teamwork-design.md

## Mode

Use mode: quick for small, direct edits.
Use mode: task for broader work that needs a clearer summary, verification,
or commit context.

## Status

Use one of:
- active
- paused
- completed
- blocked
- abandoned

## Required Sections

## Summary
One or two sentences describing the task outcome.

## Files
- path/to/changed-file

## Work
- Concise bullet describing meaningful work.
- Concise bullet describing meaningful decision or result.

## Verification
- Command, check, review, or reason verification was not run.

## Notes
- Context useful to future agents.

## Update Rules

Update the task file when:
- the task starts
- changed files become clear
- the work summary changes materially
- verification is run or intentionally skipped
- the task becomes blocked, abandoned, or ready to sync

Keep task files concise. Summarize work; keep raw chat transcripts in the CLI's
own session history.

## Sync Readiness

Before setting status: completed, make sure:
- Summary describes the outcome
- Files lists changed project files
- Work captures the important steps or decisions
- Verification is filled
- commit is present when the task produced a commit

## Safety

Keep unrelated dirty files untouched.
Ask the user when task boundaries are unclear.
Keep secrets, credentials, tokens, customer data, and private transcripts out of
task files.

Task Types

Task 类型只有两种：quick 和 task。Quick task 是快速修改， 过程更短，但仍生成 task record。普通 task 只比 quick 多一些 brief、 work summary 和 verification 要求。

  • Task 从用户和 agent client 的互动中产生。
  • Tasks tab 展示和恢复 tasks；task 创建来自 agent client 中的工作互动。
  • 一个 worktree 可以有多个 active tasks，LLM 和用户共同保持文件边界清晰。

Team Context

Team context branch 是远端同步存储；.sharkbay/team-context/ 是 SharkBay 刷新的本地 mirror。Agent 不需要 MCP，也不直接读取远端 branch； 它只需要按 protocol 搜索和阅读这个本地 mirror。 

.sharkbay/team-context/
  project.md
  tasks/
    2026/
      05/
        B4N8Q2-u8123456-m91c0a2-browser-shortcuts.md

同步后，同一个 taskId 可以同时存在于本地 .sharkbay/tasks/ 和 .sharkbay/team-context/tasks/；它们是同一个 logical task 的两个 physical copies。

Context Branch Creation

sharkbay-team-context 由 SharkBay 在用户第一次为当前 repo 启用 Teamwork 时创建。创建动作属于 Git transport 初始化，不创建 task，也不写入业务 context。

  1. 解析当前 repo 的 origin，得到 OWNER/REPO。
  2. 使用 gh 检查当前 GitHub user 和 repo permission。
  3. 写入 context branch 要求 permission 至少是 write。
  4. 检查远端是否已有 sharkbay-team-context。
  5. 如果已存在，直接 fetch 并刷新 .sharkbay/team-context/。
  6. 如果不存在，由 SharkBay 创建 branch、写入最小目录结构并 push。
  7. 如果没有权限，Teamwork sync 不启用；本地 task files 和 Tasks UI 仍可使用。
branch: sharkbay-team-context

.sharkbay-team-context/
  project.md
  tasks/
    .gitkeep
---
kind: sharkbay_team_context
repo: OWNER/REPO
createdAt: 2026-05-15T00:00:00Z
createdBy: github-login
---

Sync Loop

Teamwork 启用后，SharkBay 每分钟运行一次 sync tick。这个 loop 只做文件传输和 mirror 刷新，不修改 task 的业务内容。

Sync 只有一个触发机制：每分钟 tick。UI 展示上一次 tick 时间、pending 数量和 最近一次 transport error。

  1. Fetch sharkbay-team-context。
  2. 刷新本地只读 mirror：.sharkbay/team-context/。
  3. 扫描本地 .sharkbay/tasks/**/*.md。
  4. 找出 status: completed 且 mirror 中没有相同 taskId 的 task。
  5. 把这些 completed task files 复制到 context branch 的 tasks/YYYY/MM/。
  6. Commit and push。
  7. Push 成功后再次刷新 mirror；push rejected 时 fetch/rebase/retry。
local completed + mirror missing = pending sync
local completed + mirror exists  = synced
last transport failed            = sync failed in UI

Completed record 与本地 task file 使用同一种 Markdown 格式。Agent 完成 task 后更新 status、commit、completedAt 和正文 sections；SharkBay sync 时复制这个文件到 context branch 的日期目录。

Tasks UI Data

UI 同时读取本地 task Markdown 和 team context Markdown，但 UI 层只消费统一的 TaskViewModel。本地文件代表当前机器工作副本；team 文件代表已同步团队记录。 

local .sharkbay/tasks/**/*.md
        -> parse frontmatter + sections
team .sharkbay/team-context/tasks/**/*.md
        -> parse frontmatter + sections
        -> TaskViewModel[]
        -> Tasks UI
type TaskViewModel = {
  taskId: string
  taskTag: string
  title: string
  mode: "quick" | "task"
  status: "active" | "paused" | "completed" | "blocked" | "abandoned"
  sync: "local" | "pending" | "synced" | "failed"
  owner: { githubLogin: string; githubUserId?: number; avatarUrl?: string }
  agent?: string
  machine?: string
  createdAt?: string
  updatedAt?: string
  completedAt?: string
  files?: string[]
  summary?: string
  verification?: string
  sourceKind: "local-md" | "team-md"
  readOnly: boolean
}

Merge Rules

  • 先读本地 .sharkbay/tasks/**/*.md。
  • 再读本地 mirror 的 .sharkbay/team-context/tasks/**/*.md。
  • 按 taskId 去重。
  • sync 是 UI derived field，不写入 task Markdown。
  • local active / pending sync / failed sync 优先显示本地副本。
  • local completed 且 team copy 存在时，优先显示 team copy。
  • team exists, local missing 时显示 team copy，作为只读历史记录。

Task List

  • Tasks tab 是 Project detail 的第一个 tab，放在 Git 左边。
  • Tasks 是一体化列表，按状态和更新时间排序。
  • Teamwork 启用时左侧显示 GitHub avatar；本地模式下直接以 title 开始。
  • 每个 task 有 6 位 taskTag，例如 A7K3P9。
  • Task detail 点击后 drilldown，返回按钮回到列表。
Boundaries
  • 基础 SharkBay 可在本地使用；Teamwork sync 使用 authenticated gh。
  • Context branch 在用户第一次启用 Teamwork 时由 SharkBay 创建；已有 branch 时只 fetch。
  • 写入 context branch 要求 GitHub repo permission 至少是 write。
  • Teamwork 启用后通过每分钟 tick 自动 sync。
  • Task records 使用 compact Markdown，适合搜索、阅读和 agent context reuse。
  • 本地和团队 task records 使用同一种 Markdown 格式，frontmatter 供 UI 解析。
  • .sharkbay/team-context/ 是 SharkBay 管理的只读 mirror，供 UI 和 agent 搜索。
  • Agent 原始 session/transcript 留在对应 CLI 的存储中。
  • SharkBay 可以阻止自己提交 .sharkbay/ 和 generated root instruction files。

Architecture Risks

Risk Impact Mitigation
Instruction adherence drift Task file 缺少 summary、verification，或 changed files 需要用户复核 Tasks UI 暴露 missing fields / dirty state；commit 前由 SharkBay 展示 changed files，用户确认。
Context branch 被 force-push 或删除 团队历史需要从成员本地 task files 或 Git history 恢复 创建 sharkbay-team-context 时提示 branch protection；sync 失败必须显示。
gh token 过期或权限变化 用户以为 context 已同步，但实际失败 每次 sync 前 readiness check；失败状态只显示在 UI，不回写 task Markdown。
Root ignored instruction file 与用户文件冲突 覆盖用户自己的 AGENTS.md 等文件 只管理带 SharkBay marker 的文件；已有 tracked 或 untracked 同名文件时提示用户。

MVP Plan

  1. 实现 project-local teamwork install：写 .sharkbay/ 和本地 .git/info/exclude。
  2. 实现 Teamwork enable flow：检查 gh、repo permission，并创建或 fetch sharkbay-team-context。
  3. 生成 local instruction files 和可选 root ignored adapters。
  4. 实现 Markdown task file parser：frontmatter + sections。
  5. 实现 agent-maintained task file protocol：create/update/finish。
  6. 实现每分钟 context branch sync：只写 tasks/**/*.md，并刷新本地 mirror。
  7. 实现 Tasks UI view model：合并 local Markdown 和 .sharkbay/team-context/ Markdown。
  8. 实现 SharkBay commit guard：阻止提交 harness local files。
